# Artifact Center — 产品愿景

## 一句话定位

Artifact Center 是面向企业内网的现代化制品管理平台：用 GitHub / Linear 级别的体验，管理 APK、AAB、EXE、ZIP 等构建产物的上传、检索、分发与生命周期。

## 我们解决什么问题

企业内交付链路里，制品往往散落在：

- 共享盘与网盘文件夹
- 聊天群文件
- 临时 FTP / HTTP 静态目录
- 各业务线自建下载页

结果是：

1. **找不到**：版本命名混乱，同名文件覆盖，历史不可追溯
2. **不可信**：不知道是谁上传、对应哪次构建、是否可对外
3. **难分发**：测试、产线、售后拿包路径不统一
4. **无治理**：过期包堆积、权限粗放、审计缺失

Artifact Center 把「制品」当作一等公民：可检索、可预览元数据、可权限控制、可追溯、可扩展。

## 产品不是什么

| 不是 | 原因 |
|------|------|
| ERP / OA 后台 | 不做组织审批流大杂烩，不做信息密度竞赛 |
| 通用网盘 | 不是文件夹树 + 上传下载；是面向版本化制品的专业对象模型 |
| CI 本身 | 不替代 Jenkins / GitLab CI；可以接收 CI 推送的产物 |
| 应用商店 | 不做消费者评分、评论、推荐流（内网分发场景另议） |
| Dashboard 驾驶舱 | 首页不是 KPI 墙，是快速进入工作的入口 |

## 产品是什么

更接近这些产品的「气质与交互范式」：

- **Linear**：干净信息层级、键盘优先、状态清晰
- **GitHub**：对象（Repo/Release）可导航、元数据可读、操作克制
- **Vercel**：部署/产物状态一目了然、反馈即时
- **Raycast**：搜索即入口、少打扰
- **Apple HIG**：留白、字体层级、材质克制、动效服务理解

## 目标用户

| 角色 | 核心诉求 |
|------|----------|
| 开发工程师 | 快速上传构建产物、关联版本、发给测试 |
| 测试工程师 | 按项目/渠道/版本找包、确认元数据、下载安装 |
| 发布/运维 | 控制可见范围、归档旧包、审计下载 |
| 项目经理 / 产线支持 | 不靠问人就能拿到正确版本 |
| 平台管理员 | 权限、存储、保留策略、系统健康 |

## 核心价值主张

1. **找得到**：搜索与筛选优于层层目录
2. **信得过**：上传者、时间、校验、构建信息可追溯
3. **拿得到**：权限清晰，下载路径稳定
4. **管得住**：生命周期、保留策略、审计
5. **用得爽**：现代 SaaS 体验，而不是内网「能用就行」

## 设计原则（产品层）

1. **对象优先**：Artifact 是中心，不是「菜单 + 表格」
2. **搜索优先**：全局搜索是一级能力
3. **默认简洁**：列表只展示决策所需信息，详情再展开
4. **速度即体验**：首屏轻、操作反馈即时
5. **可扩展类型**：先 APK/AAB/EXE/ZIP，架构预留 IPA / Firmware
6. **内网可信**：权限、审计、离线部署友好

## 成功标准（方向性）

- 用户能在 **30 秒内** 找到目标制品并完成下载
- 新用户无需培训即可完成「上传 → 标注 → 分享链接」
- 测试反馈「比网盘清晰」，研发反馈「比自建页省心」
- UI 在内网演示中被评价为「像正儿八经的产品」，而非「又一个后台」

## 长期愿景

成为企业内网的 **制品基础设施**：

- 对接 CI：构建结束自动入库
- 对接通知：飞书/企业微信/邮件（可配置）
- 对接设备：扫码安装、内测分发
- 对接治理：保留策略、合规导出、签名信息展示

先把「上传、管理、查找、下载」做到极致，再扩展生态。
