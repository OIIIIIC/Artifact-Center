# Artifact Center — 产品需求文档（PRD）

> 状态：Foundation v1  
> 范围：产品定义与 MVP 边界（不含实现代码）

---

## 1. 背景与目标

### 1.1 背景

企业内部多端构建产物（Android / Windows / 通用压缩包等）缺乏统一、可信、可检索的管理入口。现有方案体验落后、元数据弱、权限与审计不足。

### 1.2 产品目标

建设 **Artifact Center**：企业内网制品中心，提供现代化的上传、检索、详情、分发与治理能力。

### 1.3 非目标（当前阶段）

- 不替代源码托管与 CI 流水线
- 不做完整 MDM / 设备管理
- 不做公网应用商店能力
- 不做复杂审批流引擎（可后续作为可选工作流）
- 不做「大数据可视化大屏」

---

## 2. 用户与场景

### 2.1 用户画像

| 角色 | 频率 | 关键路径 |
|------|------|----------|
| 开发 | 高 | 上传 → 填写元数据 → 复制链接 |
| 测试 | 高 | 搜索 → 筛选 → 下载 |
| 运维/发布 | 中 | 归档、权限、清理 |
| 管理员 | 低 | 项目、成员、存储策略 |

### 2.2 核心场景

**S1 研发上传新包**  
构建完成后上传 APK/AAB/EXE/ZIP，选择项目与环境，补充版本号/渠道/备注，获得可分享链接。

**S2 测试按条件取包**  
按项目、平台、版本、渠道、上传者、时间筛选；或全局搜索包名/版本号。

**S3 分享给协作方**  
复制短链或带权限的下载链接；接收方在权限内可下载，无权限则明确提示。

**S4 查看制品详情**  
查看文件哈希、大小、上传信息、解析出的包信息（如 APK packageName/versionCode）、历史版本关联。

**S5 治理与清理**  
标记废弃、归档、按策略删除或保留；审计谁下载了什么。

## 3. 核心概念模型

```
Organization（可选，单租户内网可弱化）
  └── Project（项目 / 产品线）
        └── Artifact（制品实体）
              ├── versions / builds（版本维度）
              ├── files（实际文件）
              ├── metadata（类型相关元数据）
              ├── tags / channels
              └── access & audit
```

### 3.1 Project

- 名称、标识（slug）、描述、可见性
- 成员与角色
- 默认保留策略（可选）

### 3.2 Artifact

制品是用户理解的「一个包 / 一次可交付物」，建议字段：

| 字段 | 说明 |
|------|------|
| id | 唯一标识 |
| projectId | 所属项目 |
| name | 展示名 |
| type | apk / aab / exe / zip / … |
| versionName | 如 1.2.3 |
| versionCode | 可选数值版本 |
| channel | 渠道 / 变体（official, debug, store…） |
| environment | dev / staging / prod 等 |
| filename | 原始文件名 |
| size | 字节 |
| checksum | SHA-256 等 |
| uploader | 上传者 |
| createdAt | 上传时间 |
| status | active / deprecated / archived |
| notes | 备注 |
| buildInfo | CI 构建号、commit、分支（可选） |
| parsedMeta | 类型解析结果（APK 包名等） |

### 3.3 类型扩展原则

每种类型有：

- 允许的扩展名 / MIME
- 可选解析器（APK 解析 packageName）
- 详情页展示模板
- 图标与标签色（克制，语义化）

新增类型（IPA、Firmware）应通过注册表扩展，而不是改核心表结构大爆炸。

## 4. 功能需求

### 4.1 MVP（P0）

| ID | 功能 | 描述 |
|----|------|------|
| F01 | 登录 | 内网账号体系（先本地/OIDC/LDAP 可配置占位） |
| F02 | 项目列表 | 卡片式项目入口，非密表格 |
| F03 | 创建/编辑项目 | 基础信息与成员（简化） |
| F04 | 上传制品 | 拖拽/选择文件，进度反馈，类型识别 |
| F05 | 元数据表单 | 版本、渠道、环境、备注；智能预填文件名 |
| F06 | 制品列表 | 卡片或紧凑行列表；强筛选 + 排序 |
| F07 | 全局搜索 | 名称、版本、文件名、备注 |
| F08 | 制品详情 | 元数据、下载、复制链接、哈希 |
| F09 | 下载 | 鉴权后下载；记录审计 |
| F10 | 状态操作 | 废弃 / 归档（软状态） |
| F11 | 基础权限 | 项目内角色：Admin / Maintainer / Viewer |
| F12 | 空状态与错误 | 专业空状态，无「系统异常请联系管理员」糊墙 |

### 4.2 增强（P1）

- CI API 上传（Token）
- 批量上传
- 版本对比（同项目同类型相邻版本）
- 扫码下载页（移动端友好）
- Webhook 通知
- 存储用量与保留策略
- APK/AAB 元数据自动解析
- 下载次数与最近下载者

### 4.3 远期（P2）

- IPA / Firmware 类型
- 签名与证书信息展示
- 灰度分发 / 过期链接
- 与设备管理、自动化测试平台集成
- 多租户

## 5. 信息架构（IA）

### 5.1 主导航（建议）

```
┌─────────────────────────────────────┐
│  Logo    Search…          User      │
├─────────────────────────────────────┤
│  Projects                           │
│  All Artifacts（跨项目，权限内）      │
│  Uploads / Activity（可选）          │
│  Settings（管理员）                   │
└─────────────────────────────────────┘
```

### 5.2 关键页面

| 页面 | 用途 | 设计要点 |
|------|------|----------|
| 登录 | 进入系统 | 极简，品牌清晰 |
| 项目列表 | 选项目 | 卡片网格，大标题 |
| 项目概览 | 最近制品 + 快捷上传 | **不是 KPI Dashboard** |
| 制品列表 | 浏览/筛选 | 工具条 + 列表/卡片 |
| 上传 | 专注流程 | 向导式，步骤少 |
| 制品详情 | 决策与下载 | 左主信息右操作 |
| 设置 | 项目/系统 | 分组清晰，低噪声 |
| 搜索结果 | 全局结果 | 按类型/项目分组 |

### 5.3 明确不做的页面形态

- 满屏多列表格 + 左侧 20 项菜单
- 首页六色图表墙
- 弹窗套弹窗的表单地狱

## 6. 权限模型（简版）

| 角色 | 项目设置 | 上传 | 编辑元数据 | 下载 | 废弃/归档 | 删除 |
|------|----------|------|------------|------|-----------|------|
| Viewer | - | - | - | ✓ | - | - |
| Maintainer | - | ✓ | ✓ | ✓ | ✓ | 软删可选 |
| Admin | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ |

系统级 Super Admin 管理全局配置与存储。

链接分享：默认需登录；P1 可支持时效 Token 链接。

## 7. 非功能需求

| 类别 | 要求 |
|------|------|
| 性能 | 列表首屏可感知 < 1s（内网、合理数据量）；上传大文件可断点/分片（P1） |
| 可用性 | 关键路径键盘可达；搜索快捷键 |
| 安全 | 鉴权、鉴权下载、审计日志、防目录遍历、病毒扫描可插拔（P2） |
| 可部署 | Docker / 内网单机或集群；对象存储可配置（本地 / MinIO / S3 兼容） |
| 可观测 | 基础健康检查、错误日志 |
| 无障碍 | 对比度达标、焦点可见、语义化结构 |
| i18n | 首版中文；文案键化预留英文 |

---

## 8. 验收标准（MVP）

1. 可创建项目并邀请/分配成员（或管理员直接加）
2. 可上传 APK、AAB、EXE、ZIP，类型正确识别
3. 列表可按项目、类型、状态筛选，可搜索版本号
4. 详情可下载，SHA-256 可见
5. Viewer 无法上传；无权限项目不可见
6. 废弃状态在列表中语义清晰且默认可过滤
7. UI 符合 `02-DESIGN.md` / `07-UI-PRINCIPLES.md`，无明显 ERP 风格

---

## 9. 风险与依赖

| 风险 | 缓解 |
|------|------|
| 大文件上传失败 | 分片、重试、清晰进度与失败恢复 |
| 存储成本飙升 | 保留策略、归档冷存储（P1） |
| 元数据不规范 | 上传引导 + 项目级命名约定 |
| 权限过粗 | 先项目级角色，再文件级（如需） |
| 解析器不可靠 | 解析失败不阻断上传，仅提示 |
