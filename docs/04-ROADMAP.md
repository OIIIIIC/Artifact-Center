# Artifact Center — 产品路线图

> 原则：先体验与核心闭环，再集成与治理。  
> 在 Foundation 文档完成前，不进入应用开发。

---

## 0. 阶段总览

| 阶段 | 名称 | 目标 | 状态 |
|------|------|------|------|
| P0 | Foundation | 规范、结构、共识 | **进行中** |
| P1 | MVP | 可内测的上传/检索/下载闭环 | 未开始 |
| P2 | Usability | 解析、CI、分享、移动下载体验 | 未开始 |
| P3 | Governance | 保留策略、审计增强、用量 | 未开始 |
| P4 | Ecosystem | 更多类型、Webhook、生态集成 | 未开始 |

---

## 1. P0 — Foundation（当前）

### 目标

建立所有人（含 AI）必须遵守的规范基线。

### 交付

- [x] 目录：`docs/` `prompts/` `references/` `assets/`
- [x] 完整文档 00–07（Foundation v1 已写入）
- [ ] 团队/个人评审：愿景、DESIGN、UI 原则无歧义
- [ ] （可选）品牌色与 Logo 方向确定，资源放入 `assets/`

### 退出标准

- 不写业务代码也能讲清产品是什么、长什么样、怎么分层
- 任意新对话可仅靠 `docs/` 对齐实现

### 明确不做

- 不创建 React/Vite 工程
- 不安装依赖
- 不写页面

---

## 2. P1 — MVP

### 目标

内网可用的最小闭环：**登录 → 项目 → 上传 → 列表/搜索 → 详情 → 下载**。

### 范围

**产品**

- 项目 CRUD（简化）
- 成员角色：Admin / Maintainer / Viewer
- 制品上传：APK / AAB / EXE / ZIP
- 元数据：版本、渠道、环境、备注
- 列表筛选 + 搜索
- 详情下载 + 哈希展示
- 废弃/归档状态
- 基础审计（上传/下载）

**体验**

- 符合 DESIGN 的壳布局、项目卡、制品列表、上传流、详情页
- 完整空状态 / 错误态
- 非 Dashboard 首页

**工程**

- Web + API + DB + 本地或 MinIO 存储
- Docker Compose 可启动（建议）

### 退出标准

- 真实项目组完成一次完整周迭代试用
- PRD MVP 验收项全部通过
- DESIGN 检查清单无 P0 违规

### 不做（MVP 砍掉）

- CI Token、Webhook
- APK 深度解析
- 多文件版本树复杂模型
- 复杂报表
- IPA / Firmware

---

## 3. P2 — Usability

### 目标

减少「填表负担」和「分发摩擦」。

### 候选能力

- 文件名智能推断版本
- APK/AAB 基础元数据解析
- CI API 上传 + Token
- 复制分享链接体验打磨
- 移动端详情/下载页
- 全局 Command Palette 搜索
- 上传分片与断点（大文件）
- 活动流（谁上传了什么）

### 退出标准

- CI 可自动入库
- 测试同学主要路径不求人

---

## 4. P3 — Governance

### 目标

让平台可长期运营，而非只是「好用的网盘」。

### 候选能力

- 项目级保留策略（N 天 / 每渠道保留最近 M 个）
- 存储用量统计（设置内，非首页大盘）
- 审计导出
- 强制归档工作流
- 病毒扫描插件位
- SSO（OIDC/LDAP）生产加固

---

## 5. P4 — Ecosystem

### 目标

成为内网制品基础设施。

### 候选能力

- IPA、Firmware 等类型
- Webhook / 通知集成
- 与测试平台、设备云集成
- 灰度链接、过期链接
- 多仓/多区域存储
- 开放更丰富的 Open API

---

## 6. 优先级原则

1. **用户闭环 > 技术炫技**  
2. **体验规范 > 功能堆砌**  
3. **一种类型做深 > 十种类型做浅**（MVP 四种文件类型但流程同一套）  
4. **可部署可备份 > 微服务面子**  
5. **有度量再优化**：先能用，再谈上传加速等
