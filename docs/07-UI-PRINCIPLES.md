# Artifact Center — UI 原则

> 比组件更底层的判断标准。  
> 当没有现成组件规范时，用本文做决策。

---

## 1. 第一性原理

### 1.1 界面是决策工具

用户来这里是为了：

**找到正确的包 → 确认可信 → 下载或分享**

任何不服务这条链路的 UI 元素，默认删除。

### 1.2 清晰优于完备

一屏只回答有限问题。  
次要信息进入详情、菜单或「更多」。

### 1.3 安静的专业感

Premium 来自对齐、留白、字体与克制，  
不来自装饰、渐变和动效表演。

### 1.4 速度是 UI 的一部分

快的骨架、即时的按钮反馈、不卡的搜索，  
比多一个图表更像现代产品。

---

## 2. 十条军规

1. **一个页面一个主要任务**  
2. **一个区域一个主按钮**  
3. **默认视图保持简洁**（高级密度可选）  
4. **搜索是一级公民**  
5. **对象用卡片/优雅列表，不用 Excel 墙**  
6. **中性色为主，颜色只表达语义**  
7. **边框分区，阴影让位**  
8. **动效解释变化，不喧宾夺主**  
9. **空状态也是产品**  
10. **文案像同事说话，不像公文系统**  

---

## 3. 信息层级

### 3.1 三级信息

| 级别 | 例子 | 视觉 |
|------|------|------|
| L1 | 页面标题、文件名 | 大号、高对比 |
| L2 | 版本、类型、主操作 | 中号、明确 |
| L3 | 时间、上传者、哈希缩略 | 小号、弱对比 |

错误层级是 UI 廉价感的来源：所有字一样大、一样黑。

### 3.2 扫描模式

列表页按 **F 型 / 左标题右状态** 扫读设计：

- 左：名称与版本  
- 中：类型/环境  
- 右：时间与操作  

不要让眼睛在行内反复横跳超过必要次数。

---

## 4. 导航原则

- 导航项少而稳，不随功能膨胀无限加菜单  
- 用户应始终知道：我在哪、如何返回、主操作在哪  
- 面包屑轻量可选；不要又长又臭  
- 项目上下文清晰（当前 Project 名称可见）  

---

## 5. 反馈原则

| 用户动作 | 系统反馈 |
|----------|----------|
| 点击 | 100ms 内可视状态变化 |
| 提交 | 按钮 loading，防重复 |
| 成功 | toast 或内联，可接下一步 |
| 失败 | 说人话 + 怎么办 |
| 长任务 | 进度可感知，可取消（若可行） |

无反馈 = 劣质。  
过度反馈（狂弹窗）= 劣质。

---

## 6. 表单原则

- 字段越少越好；能推断的预填  
- 相关字段分组，组间留白  
- 先选文件再填元数据（上传流）  
- 校验即时且局部  
- 绝不在提交后才告知「缺了 12 个字段」而无定位  

---

## 7. 列表与密度

### 7.1 默认

Comfortable：可点、可读、可呼吸。

### 7.2 何时用表

仅当用户需要 **多列对比**（如运维审计）且字段 ≥ 需要对齐时。  
即使使用 Table，也要：

- 限列  
- 舒适行高  
- 去掉视觉噪音  

### 7.3 何时用卡

项目、摘要对象、强调「实体感」时。

---

## 8. 权限与 UI

- 无权限：隐藏 vs 禁用要有策略  
  - 完全不可见的资源：不出现  
  - 可知但不可用：禁用 + 原因  
- 不要让用户点到最后 403 才知道  
- 管理员与成员看到的信息架构一致，能力不同  

---

## 9. 信任设计

制品平台的 UI 必须传递 **可信**：

- 展示校验和、上传者、时间  
- 状态语义准确（deprecated 不是 active）  
- 危险操作二次确认且可理解后果  
- 链接分享不制造虚假「已公开」错觉  

信任靠细节，不靠「安全认证」徽章贴纸。

---

## 10. 动效哲学

问三个问题：

1. 它帮助理解状态吗？  
2. 它让界面显得更慢吗？  
3. 关掉动画后是否仍完整？  

只有 1=是、2=否、3=是 时保留。

---

## 11. 反模式速查

| 看到这个 | 判定 |
|----------|------|
| 首页六张统计卡 + 图表 | 拒绝 |
| 左侧 15 个一级菜单 | 拒绝 |
| 默认满屏 ProTable | 拒绝 |
| 五颜六色标签行 | 拒绝 |
| 重阴影悬浮卡瀑布 | 拒绝 |
| 成功撒花 | 拒绝 |
| 「系统繁忙请稍后再试」无 requestId/无指引 | 拒绝 |
| 为对齐 ERP 习惯而加密度 | 拒绝 |

---

## 12. 决策流程图

```
新 UI 需求
  → 是否服务「找包/信包/取包」？ 否 → 砍掉或降级
  → 是否可用现有页面模式容纳？ 是 → 扩展，不新开菜单
  → 是否符合 DESIGN 关键词？ 否 → 重做视觉
  → 是否引入第二种视觉语言？ 是 → 拒绝
  → 实现
```

---

## 13. 评审用语（团队对齐）

鼓励在评审中直接说：

- 「太密了，违反 Large Spacing」  
- 「这是 ERP 气质，不符合愿景」  
- 「主按钮不唯一」  
- 「这页没有空状态」  
- 「颜色在装饰而不是语义」  

不鼓励：

- 「先这样，以后再美化」（对 P0 视觉债务零容忍）

---

## 15. 结语

好的 UI 原则不是海报，而是 **否决权**。

当功能与气质冲突时，  
**保护气质，重新设计功能表达**——  
这正是 Artifact Center 与普通内网后台的差别。
