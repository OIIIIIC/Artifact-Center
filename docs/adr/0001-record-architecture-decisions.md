# ADR-0001: 采用 ADR 记录架构决策

- **状态**: accepted
- **日期**: 2025-07-19
- **决策者**: artifact-center 开发团队

---

## 背景与问题陈述

artifact-center 项目从 P0 Foundation 阶段进入 P1 MVP 阶段，技术选型和架构设计逐步复杂化。当前架构决策散落于 `docs/` 目录下的规范文档中，缺乏标准化的、可追溯的决策记录机制。随着团队协作和后续迭代，若没有统一的决策日志，新成员难以理解历史选型动机，已有决策也缺乏复审和废弃的明确路径。

## 决策

我们将在 `docs/adr/` 目录下采用 **Architecture Decision Records (ADR)** 机制记录所有关键架构决策。

具体规则：

- 使用 Michael Nygard 提出的 ADR 模板（背景 → 决策 → 替代方案 → 后果）
- 文件编号采用四位数字 (0001, 0002, ...)
- 每篇 ADR 标注状态：`proposed` / `accepted` / `deprecated` / `superseded`
- 新 ADR 从 `template.md` 复制起始

## 考虑的替代方案

### 替代方案 A: 在 AGENTS.md 或规范文档中内联决策

- **优点**: 零额外文件，查找方便
- **缺点**: 决策与产品规范耦合，缺乏独立的生命周期管理；决策内容容易被冗长的规范淹没；不利于按时间线追溯决策演变
- **为何未采纳**: 随着决策数量增加，内联方式会显著降低可维护性

### 替代方案 B: 使用 Notion / Confluence 等外部 Wiki

- **优点**: 协作体验好，支持富文本
- **缺点**: 与代码仓库分离，版本控制困难；离线不可用；依赖第三方服务
- **为何未采纳**: 决策应随代码一同版本管理，Git 仓库内的 Markdown 是最可靠的选择

## 后果

### 正面影响

- 每个重要技术选型都有独立的、可追溯的记录
- 新成员可通过 ADR 目录快速了解项目历史决策
- 决策复审和废弃有明确流程

### 需要关注

- 团队需养成"重要决策写 ADR"的习惯，初期可能遗漏
- ADR 数量增长后需定期清理废弃条目
- 需确保 ADR 的标题和编号不冲突
