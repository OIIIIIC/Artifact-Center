# ADR-0006: 搜索能力渐进演进策略

- **状态**: accepted
- **日期**: 2026-07-19
- **决策者**: 产品规范 (03-ARCHITECTURE.md §8, 10-DATABASE.md §4)

## 背景

Artifact Center 需要支持按名称、版本号、文件名、备注等维度的全局搜索。"搜索优先"是产品体验的核心承诺。但 MVP 阶段数据量有限，不应过早引入重量级搜索引擎。

## 决策

采用三阶段渐进策略，**接口形状先稳定**：

| 阶段   | 方案                                  | 触发条件           |
| ------ | ------------------------------------- | ------------------ |
| MVP    | PostgreSQL `ILIKE` + 合理 B-tree 索引 | 当前               |
| 成长   | 物化搜索字段 / `pg_trgm` GIN 索引     | ILIKE 性能不可接受 |
| 大规模 | OpenSearch / Elasticsearch            | pg_trgm 仍不足     |

关键约束：**不要在数据量很小时预先建立大量 GIN 索引**。

## 替代方案

- **首版即上 Elasticsearch**: MVP 阶段运维负担过大(Docker Compose 内网单机部署)，且数据量未到需要 ES 的程度
- **仅做精确匹配**: 产品要求"搜索优先"，严格的精确匹配不满足用户期望

## 后果

- **正面**: MVP 部署简单(仅 PostgreSQL)，随着数据增长可平滑升级
- **负面**: 切换到 ES 时搜索接口实现需要重写(但 API 契约可保持不变)
- **约束**: API 层面搜索接口设计需考虑未来替换实现的灵活性
